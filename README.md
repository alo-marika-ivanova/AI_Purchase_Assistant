# AI Purchase Assistant

A lightweight Streamlit purchasing assistant for one internal buyer. It supports simulated negotiations and real email/WhatsApp communication using the same case workflow.

## Preserved functionality

- Supplier selection and purchasing case creation
- Per-case communication mode:
  - **Simulation:** outbound messages remain in Streamlit; supplier replies are entered manually
  - **Real:** outbound messages use each supplier's configured email or WhatsApp channel
- Personalized buyer messages generated by local Ollama
- Supplier-specific deterministic fallbacks if Ollama is unavailable or returns malformed text
- Initial RFQs and RFQ reminders
- Email threading and Microsoft Graph reply import
- WhatsApp Cloud API outbound messages and inbound webhook handling
- Local semantic extraction of common supplier replies
- One clarification request for unclear RFQ pricing
- Human-review pause for risky, compound, or unusual replies
- Multi-supplier offer comparison
- Negotiation target based on the configured discount percentage
- Explicit price-reduction requests to suppliers
- Common negotiation handling:
  - target acceptance
  - improved counteroffer
  - unchanged/final price
  - refusal to reduce
  - supplier will reply later
- Winner recommendation by lowest confirmed price
- Manual winner selection and winner notification

## Communication mode

The checkbox **Send real messages for this case** is stored in `negotiation_cases.auto_send_messages` and is the single source of truth.

- Unchecked: all automatic and manual buyer messages are simulated in chat.
- Checked: all buyer messages use the supplier's configured contact channel.

Environment variables no longer switch an individual case between simulation and real delivery.

## Setup

### 1. Create environment

```powershell
python -m venv .venv
.\.venv\Scripts\Activate.ps1
python -m pip install -r requirements.txt
```

### 2. Configure credentials

```powershell
Copy-Item .env.example .env
notepad .env
```

Do not commit `.env`.

### 3. Configure suppliers

Edit:

```text
data/suppliers.csv
```

Real cases require valid contact data. Addresses ending in `.example` are suitable only for simulation.

### 4. Initialize database

```powershell
python .\scripts\init_db.py
```

### 5. Run the deterministic presentation test

```powershell
python .\scripts\test_presentation_workflow.py
```

Expected result:

```text
PRESENTATION WORKFLOW TEST PASSED
```

This test uses a temporary database and mocked transports. It does not send real messages or alter the normal project database.

### 6. Start Streamlit

```powershell
streamlit run ui\streamlit_app_clean.py
```

### 7. Start the workflow worker

```powershell
python .\scripts\email_worker.py
```

The worker is required for time-based RFQ reminders and automatic processing of active cases. Simulation cases do not call Microsoft Graph.

### 8. Start the WhatsApp webhook when needed

```powershell
uvicorn app.main:app --host 0.0.0.0 --port 8000 --reload
```

## Presentation workflow

1. Create a case and select two suppliers.
2. Leave **Send real messages for this case** unchecked for simulation.
3. Start negotiating.
4. RFQs appear in each supplier chat with supplier-specific wording.
5. Enter supplier replies manually, for example:
   - `Our price is 42 USD per unit.`
   - `Our price is 50 USD per unit.`
6. After the minimum number of offers is reached, the system compares offers, calculates a target, and immediately generates price-reduction requests.
7. Enter negotiation replies, for example:
   - `Yes, we can do that.`
   - `I can go to 39 USD, but that is my final offer.`
8. Review the suggested winner and press the supplier's notification button.

For real communication, create a separate case with the checkbox checked and use suppliers with valid credentials/contact data.

## Negotiation policy

Testing and production timings are configured in:

```text
config/negotiation_policy.json
```

The presentation configuration uses short RFQ reminder timing. Change `mode` to `production` before deployment testing.

## Cleaned obsolete components

The following were removed because they were not part of the active Streamlit/worker/webhook path:

- old negotiation engines
- duplicate message/RFQ services
- old approval-mode services
- experimental V2 semantic analyzers
- one-off database repair and progression patch scripts
- obsolete extractor tests and migration scripts

The active personalized communication writer, local classifier, common-reply policy, transport adapters, workflow planners, and UI were retained.
